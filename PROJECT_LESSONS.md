# [Youngil_WebApps 기록]

명절선물뽑기.html, 사다리타기.html, 테트리스.html 를 개발하면서 있었던 실수 및 재발방지 기록이다.

이 문서는 `Youngil_WebApps` 프로젝트에서 개별 HTML 웹페이지를 만들 때 반복했던 시행착오를 한 곳에 모아두기 위한 기록이다.  
각 웹페이지는 독립 실행형 HTML 파일이지만, 실제 개발 과정에서는 공통적으로 레이아웃, 모바일 화면 대응, 버튼 이벤트, JavaScript 상태 관리, CSS 우선순위, 브라우저 호환성, GitHub Pages 배포 확인 문제가 반복되었다.

앞으로 새로운 웹페이지를 추가할 때는 이 문서에 같은 방식으로 섹션을 추가한다.  
구분 기준은 파일명 단위로 하며, 각 파일 아래에 `목표`, `실패와 원인`, `재발 방지`, `검증 체크리스트`, `확정 규칙`을 기록한다.

---

## 전체 프로젝트 공통 회고

### 프로젝트명

`Youngil_WebApps`

### 대상 파일

- `명절선물뽑기.html`
- `사다리타기.html`
- `테트리스.html`

### 공통 목표

간단한 이벤트용 웹앱을 HTML 단일 파일 형태로 만들고, PC와 모바일에서 바로 실행할 수 있도록 구성한다.  
가능하면 외부 라이브러리에 의존하지 않고, HTML/CSS/JavaScript만으로 동작하게 만든다.  
GitHub Pages에 올렸을 때도 경로 문제 없이 실행되어야 한다.

### 공통적으로 반복된 문제

개별 웹페이지는 기능은 단순해 보였지만, 실제 개발에서는 다음 문제가 반복되었다.

- HTML 파일 하나 안에 구조, 스타일, 스크립트가 섞이면서 수정 위치를 놓침
- 모바일 화면에서 버튼, 입력창, 게임판, 결과 영역이 잘리거나 겹침
- JavaScript 이벤트 연결 시점이 맞지 않아 버튼이 눌리지 않음
- 변수 초기화와 상태 리셋이 제대로 되지 않아 이전 실행 결과가 남음
- CSS 선택자 충돌로 의도한 디자인이 적용되지 않음
- 한글 파일명과 GitHub Pages 경로 문제를 충분히 검증하지 않음
- PC에서는 정상인데 모바일 브라우저에서는 터치, 키보드, 스크롤 동작이 다르게 나타남
- AI가 “수정 완료”라고 했지만 실제 브라우저 테스트 없이 넘어가려는 문제가 있었음

### 공통 재발 방지 원칙

- 단일 HTML이라도 `HTML 영역`, `CSS 영역`, `JavaScript 영역`을 주석으로 명확히 구분한다.
- 버튼 ID, 클래스명, 함수명을 한 번 정하면 중간에 임의로 바꾸지 않는다.
- JavaScript 수정 후에는 반드시 브라우저 콘솔 오류를 확인한다.
- 실행 버튼, 리셋 버튼, 다시하기 버튼은 반드시 반복 클릭 테스트를 한다.
- 모바일 화면 기준으로 먼저 레이아웃을 확인한 뒤 PC 화면을 보정한다.
- GitHub Pages 배포 후 실제 URL에서 새로고침, 캐시 제거, 모바일 접속까지 확인한다.
- 결과 화면이 있는 웹앱은 “첫 실행”, “재실행”, “리셋 후 실행”을 모두 테스트한다.
- 게임형 웹앱은 시작, 일시정지, 재시작, 게임오버, 점수 초기화까지 별도 테스트한다.

# 1. 명절선물뽑기.html

## 작업 회고

### 목표

명절 선물 대상자를 재미있게 뽑을 수 있는 간단한 추첨 웹페이지를 만든다.  
사용자는 이름 또는 항목을 입력하고, 버튼을 눌러 무작위로 당첨자 또는 선물을 뽑을 수 있어야 한다.  
결과는 한눈에 보기 좋게 표시되어야 하며, 행사나 회사 내부 이벤트에서 바로 사용할 수 있어야 한다.

## 실패와 원인

### 1. 입력값 검증 없이 추첨 로직을 먼저 구현함

초기에는 추첨 버튼을 누르면 바로 랜덤 선택이 실행되도록 구성했지만, 입력값이 비어 있거나 줄바꿈만 있는 경우에도 실행될 수 있었다.  
그 결과 빈 항목이 당첨자로 표시되거나, 결과 영역이 이상하게 보이는 문제가 생길 수 있었다.

재발 방지:

- 추첨 실행 전 입력값을 `trim()` 처리한다.
- 빈 줄은 제외한다.
- 실제 후보 목록 개수가 1개 이상인지 확인한다.
- 후보가 없으면 추첨을 실행하지 말고 안내 메시지를 표시한다.

### 2. 줄바꿈 입력과 쉼표 입력을 구분하지 않음

사용자는 이름을 한 줄씩 입력할 수도 있고, 쉼표로 구분해서 입력할 수도 있다.  
초기 로직이 한 가지 입력 방식만 전제로 하면 실제 사용자가 다른 방식으로 입력했을 때 후보가 제대로 분리되지 않는다.

재발 방지:

- 후보 분리는 `\n`, `,`, `，` 등을 함께 고려한다.
- 분리 후 각 항목은 `trim()` 처리한다.
- 빈 문자열은 후보 목록에서 제거한다.
- 입력 예시를 화면에 표시한다.

### 3. 중복 당첨 방지 기준이 명확하지 않음

한 번 뽑힌 사람이 다시 뽑혀도 되는지, 아니면 제외해야 하는지 기준이 명확하지 않으면 추첨 결과가 신뢰를 잃는다.  
행사용 추첨에서는 대부분 중복 당첨 방지가 필요하다.

재발 방지:

- 중복 허용 여부를 코드 또는 UI에서 명확히 정한다.
- 기본값은 “중복 당첨 방지”로 둔다.
- 당첨된 후보는 `selectedList` 또는 `pickedSet`에 기록한다.
- 남은 후보가 없으면 추첨 버튼을 비활성화하거나 완료 메시지를 표시한다.

### 4. 결과 표시와 내부 상태가 분리되지 않음

화면에는 결과가 지워졌지만 내부 배열에는 당첨 이력이 남아 있거나, 반대로 화면에는 남아 있는데 내부 상태는 초기화되는 문제가 발생할 수 있었다.

재발 방지:

- `resetDraw()` 함수에서 화면과 내부 상태를 동시에 초기화한다.
- `candidates`, `pickedList`, `remainingList`, `currentWinner` 같은 상태 변수 역할을 명확히 구분한다.
- 화면만 지우는 함수와 전체 초기화 함수는 분리한다.
- 다시하기 버튼은 내부 상태까지 리셋하는지 확인한다.

### 5. 랜덤 애니메이션과 실제 당첨 결과가 불일치할 수 있음

추첨 효과를 위해 이름이 빠르게 바뀌는 애니메이션을 넣을 경우, 마지막에 표시된 이름과 실제 저장된 당첨자가 다르면 문제가 된다.

재발 방지:

- 실제 당첨자는 애니메이션 시작 전에 확정하거나, 애니메이션 종료 시점에 한 번만 확정한다.
- 화면에 최종 표시한 값과 내부 저장값이 같은지 검증한다.
- `setInterval`과 `setTimeout`을 함께 사용할 때는 타이머 ID를 변수로 저장하고 종료 시 반드시 해제한다.

### 6. 모바일에서 버튼과 결과 영역이 과도하게 커지거나 잘림

행사용 페이지는 PC보다 모바일에서 바로 열어보는 경우가 많다.  
하지만 큰 제목, 큰 버튼, 긴 결과 문자열이 있을 때 화면 밖으로 밀리거나 레이아웃이 깨질 수 있다.

재발 방지:

- 전체 컨테이너에 `max-width`를 둔다.
- 긴 이름은 `word-break: keep-all` 또는 `overflow-wrap`을 적절히 사용한다.
- 버튼은 모바일에서 터치하기 충분한 높이를 주되, 화면 전체를 차지하지 않게 한다.
- `@media` 쿼리로 작은 화면 전용 크기를 조정한다.

## 확정된 실행 규칙

- 후보 입력값은 반드시 정제 후 사용한다.
- 빈 줄과 공백만 있는 항목은 후보에서 제외한다.
- 중복 당첨 방지 여부는 명확히 관리한다.
- 추첨 결과와 내부 상태는 항상 일치해야 한다.
- 다시하기는 화면과 내부 배열을 모두 초기화해야 한다.
- 랜덤 애니메이션을 넣을 경우 타이머 중복 실행을 막는다.
- 모바일 화면에서 결과가 잘리지 않는지 반드시 확인한다.

# 2. 사다리타기.html

## 작업 회고

### 목표

참가자와 결과 항목을 입력하면 사다리타기 방식으로 매칭 결과를 보여주는 웹페이지를 만든다.  
참가자 수에 따라 사다리 세로줄과 가로줄이 동적으로 생성되어야 하며, 사용자는 결과가 자연스럽게 연결되는 과정을 확인할 수 있어야 한다.

## 실패와 원인

### 1. 참가자 수와 결과 수가 다를 때 처리가 부족함

사다리타기는 참가자 수와 결과 수가 맞지 않으면 매칭이 애매해진다.  
초기에는 이 조건을 강하게 검증하지 않으면 배열 인덱스가 어긋나거나 결과가 `undefined`로 표시될 수 있었다.

재발 방지:

- 사다리 생성 전 참가자 수와 결과 수가 같은지 확인한다.
- 수가 다르면 사다리 생성을 중단하고 안내한다.
- 최소 참가자 수 기준을 둔다.
- 입력값 정제 후 실제 개수 기준으로 검증한다.

### 2. 사다리 선 생성 로직이 시각적 위치와 내부 데이터 구조를 혼동함

사다리타기는 화면에 보이는 가로선과 내부 경로 계산용 데이터가 일치해야 한다.  
가로선을 그렸지만 내부 배열에 반영되지 않거나, 내부 배열에는 있는데 화면에는 보이지 않으면 결과가 신뢰를 잃는다.

재발 방지:

- `ladder[row][col] = true` 같은 내부 구조를 먼저 만든다.
- 화면은 내부 데이터 구조를 기준으로만 그린다.
- 시각적 CSS 위치 계산과 결과 계산 로직을 분리한다.
- 그려진 선 개수와 내부 가로선 개수를 디버그 로그로 비교할 수 있게 한다.

### 3. 인접 가로선 중복 방지 처리가 부족함

사다리타기에서 같은 높이에 인접한 가로선이 연속으로 생기면 경로가 비정상적으로 보이거나 사용자가 이해하기 어렵다.  
초기 랜덤 생성 방식에서는 가로선이 너무 몰리거나 서로 붙는 문제가 생길 수 있었다.

재발 방지:

- 같은 row에서 `col`, `col - 1`, `col + 1`에 이미 가로선이 있는지 확인한다.
- 가로선 생성 확률을 너무 높게 잡지 않는다.
- 참가자 수가 적을 때와 많을 때 생성 밀도를 다르게 조정한다.
- 무작위 생성 후 최소/최대 가로선 수를 검증한다.

### 4. 경로 추적 애니메이션이 결과 계산과 다르게 움직일 수 있음

사용자가 특정 참가자를 눌렀을 때 이동 경로를 애니메이션으로 보여주는 경우, 내부 계산 결과와 화면 이동 경로가 다르면 치명적이다.  
특히 가로 이동 후 세로 이동 순서, 세로 이동 중 가로선 감지 위치가 어긋날 수 있다.

재발 방지:

- 결과 계산 함수와 애니메이션 경로 생성 함수가 같은 내부 데이터 구조를 사용해야 한다.
- 애니메이션 전 `calculateResult(startIndex)`로 최종 결과를 먼저 계산한다.
- 애니메이션 종료 위치와 계산 결과 인덱스가 같은지 검증한다.
- 클릭 중복을 막기 위해 애니메이션 진행 중에는 버튼을 잠근다.

### 5. Canvas 방식과 DOM 방식의 기준이 섞일 수 있음

사다리 선을 `canvas`로 그릴지, `div` 요소로 만들지에 따라 좌표 계산 방식이 달라진다.  
둘을 섞으면 화면 크기 변경 시 선 위치가 맞지 않는 문제가 생길 수 있다.

재발 방지:

- 한 파일 안에서는 Canvas 방식 또는 DOM 방식 중 하나를 기본 방식으로 정한다.
- Canvas를 쓰면 `resize` 시 다시 그린다.
- DOM을 쓰면 컨테이너 기준 상대 위치를 사용한다.
- CSS transform이 좌표 계산에 영향을 주는지 확인한다.

### 6. 모바일에서 가로폭 부족으로 참가자 이름과 사다리가 겹침

참가자 수가 늘어나면 세로줄 간격이 줄어들고, 이름 텍스트가 겹칠 수 있다.  
모바일에서는 특히 사다리 영역이 좁아져 결과 확인이 어려워진다.

재발 방지:

- 참가자 수가 많을 때는 가로 스크롤을 허용한다.
- 사다리 영역에 `min-width`를 설정한다.
- 모바일에서는 이름 글자 크기를 줄이거나 세로 배치한다.
- 결과 영역은 사다리 아래에 별도로 표시한다.

## 확정된 실행 규칙

- 참가자 수와 결과 수는 반드시 같아야 한다.
- 내부 사다리 데이터 구조를 기준으로 화면을 그린다.
- 가로선은 인접 중복을 피해서 생성한다.
- 경로 계산과 애니메이션은 같은 데이터를 사용한다.
- 애니메이션 중에는 재클릭을 막는다.

- 결과는 애니메이션과 별도로 최종 텍스트로도 표시한다.

# 3. 테트리스.html

## 작업 회고

### 목표

브라우저에서 실행되는 간단한 테트리스 게임을 단일 HTML 파일로 만든다.  
블록 생성, 이동, 회전, 하강, 라인 삭제, 점수 계산, 게임오버, 재시작 기능이 정상 동작해야 한다.  
PC 키보드 조작과 모바일 버튼 조작을 모두 고려한다.

## 실패와 원인

### 1. 게임판 배열과 화면 렌더링이 분리되지 않음

테트리스는 내부 보드 배열과 화면에 보이는 셀이 정확히 일치해야 한다.  
초기 구조에서 보드 배열, 현재 블록, 고정 블록, 미리보기 블록이 섞이면 화면에는 남아 있는데 내부에는 없거나, 내부에는 있는데 화면에는 안 보이는 문제가 발생할 수 있었다.

재발 방지:

- `board`는 고정된 블록만 저장한다.
- 현재 떨어지는 블록은 `currentPiece`로 따로 관리한다.
- 렌더링 시 `board + currentPiece`를 합쳐서 화면에 표시한다.
- 블록이 바닥에 닿으면 그때만 `mergePiece()`로 board에 반영한다.

### 2. 충돌 검사 기준이 불명확함

블록 이동과 회전에서 가장 중요한 것은 충돌 검사다.  
좌우 벽, 바닥, 기존 블록과의 충돌을 정확히 검사하지 않으면 블록이 벽을 뚫거나, 기존 블록과 겹치거나, 갑자기 게임오버가 발생할 수 있다.

재발 방지:

- 모든 이동과 회전은 먼저 가상 위치를 만든 뒤 `isValidMove()`로 검사한다.
- 유효하지 않으면 실제 상태를 변경하지 않는다.
- `x < 0`, `x >= COLS`, `y >= ROWS`, `board[y][x]` 조건을 분리해서 확인한다.
- `y < 0` 영역은 새 블록 생성과 게임오버 검사에서 별도 처리한다.

### 3. 회전 후 벽에 걸리는 문제를 고려하지 않음

블록이 벽 근처에서 회전할 때 단순 회전만 적용하면 유효하지 않은 위치가 되어 회전이 막힌다.  
일반 테트리스에서는 벽 차기 방식이 있지만, 간단 구현에서는 최소한 좌우 보정이 필요하다.

재발 방지:

- 회전 후 현재 위치에서 불가능하면 `x - 1`, `x + 1`, `x - 2`, `x + 2` 순서로 보정 시도한다.
- 보정 후에도 불가능하면 회전을 취소한다.
- I 블록처럼 긴 블록은 별도 확인한다.
- 회전 전후 블록 모양 배열이 원본을 오염시키지 않도록 복사본을 사용한다.

### 4. 자동 하강 타이머가 중복 실행됨

게임 시작 버튼을 여러 번 누르거나 재시작할 때 기존 `setInterval`이 해제되지 않으면 블록이 비정상적으로 빠르게 내려간다.

재발 방지:

- `gameLoopId` 또는 `dropTimer`를 전역 상태로 관리한다.
- 게임 시작 전 기존 타이머가 있으면 `clearInterval()` 한다.
- 게임오버와 재시작에서도 타이머를 반드시 해제한다.
- 시작 버튼은 게임 중에는 비활성화하거나 `isRunning` 상태로 중복 실행을 막는다.

### 5. 키보드 이벤트가 브라우저 스크롤과 충돌함

방향키와 스페이스바를 게임 조작에 사용할 때 브라우저 기본 스크롤이 함께 발생할 수 있다.  
특히 모바일 또는 작은 화면에서 페이지가 움직이면 게임 조작감이 나빠진다.

재발 방지:

- 게임 조작 키 입력 시 `event.preventDefault()`를 호출한다.
- 입력 대상이 텍스트 입력창일 때는 게임 키 입력을 무시한다.
- Space, ArrowLeft, ArrowRight, ArrowDown, ArrowUp 키를 명확히 처리한다.
- 모바일 조작 버튼에는 `touchstart` 또는 클릭 이벤트를 별도 검토한다.

### 6. 라인 삭제 후 점수와 보드 갱신 순서가 꼬임

한 줄 또는 여러 줄이 동시에 완성되었을 때 보드에서 삭제하고 새 빈 줄을 추가하는 순서가 잘못되면 블록이 사라지거나 빈 줄 위치가 잘못될 수 있다.

재발 방지:

- 라인 삭제는 아래에서 위로 검사한다.
- 완성된 줄 수를 먼저 계산하거나, 삭제하면서 인덱스를 보정한다.
- 삭제된 줄 수에 따라 점수를 계산한다.
- 보드 갱신 후 즉시 렌더링한다.

### 7. 새 블록 생성 직후 게임오버 판정이 늦음

새 블록이 생성되었을 때 이미 기존 블록과 겹친다면 즉시 게임오버가 되어야 한다.  
이 판정을 늦게 하면 화면상으로 블록이 겹쳐 보이거나 다음 타이머에서야 종료된다.

재발 방지:

- `spawnPiece()` 직후 `isValidMove()`를 검사한다.
- 유효하지 않으면 즉시 `gameOver()`를 호출한다.
- 게임오버 상태에서는 이동, 회전, 하강 입력을 모두 무시한다.
- 게임오버 메시지와 재시작 버튼 상태를 함께 갱신한다.

### 8. 모바일 조작 버튼의 연속 입력과 키보드 입력이 다르게 동작함

PC 키보드에서는 방향키 반복 입력이 자연스럽지만, 모바일 버튼은 클릭 이벤트만 쓰면 조작이 답답할 수 있다.  
반대로 길게 누르기 처리를 잘못하면 입력이 과도하게 반복될 수 있다.

재발 방지:

- 모바일 버튼은 기본 클릭 동작부터 안정화한다.
- 길게 누르기 반복 입력을 넣을 경우 시작/종료 이벤트를 명확히 구분한다.
- `touchstart`, `touchend`, `mousedown`, `mouseup`, `mouseleave`를 함께 고려한다.
- 게임오버 또는 일시정지 시 반복 입력 타이머를 해제한다.

## 확정된 실행 규칙

- `board`에는 고정 블록만 저장한다.
- 현재 블록은 별도 상태로 관리한다.
- 이동과 회전은 항상 충돌 검사 후 반영한다.
- 자동 하강 타이머는 한 번만 실행되어야 한다.
- 게임오버 시 모든 타이머와 입력을 중단한다.
- 라인 삭제 후 점수, 레벨, 화면을 순서대로 갱신한다.
- 키보드 입력은 브라우저 기본 동작과 충돌하지 않게 처리한다.
- 모바일 버튼 조작은 PC 키보드 조작과 같은 함수로 연결한다.

# 4. Youngil_WebApps 공통 확정 규칙

## 파일 구조 규칙

- 각 웹앱은 기본적으로 단일 HTML 파일로 유지한다.
- 파일 내부에는 다음 주석 구분을 둔다.

```html
<!-- =========================
     HTML Structure
========================= -->

<style>
/* =========================
   CSS Styles
========================= */
</style>

<script>
/* =========================
   JavaScript Logic
========================= */
</script>
```

- 기능이 커질 경우에도 먼저 단일 파일에서 안정화한 뒤 분리 여부를 결정한다.
- 한글 파일명은 로컬에서는 가능하지만, 배포 URL과 공유 편의성을 고려해 영문 별칭 파일을 추가할지 검토한다.

## UI/UX 규칙

- 모바일에서 먼저 확인한다.
- 버튼은 터치 가능한 크기로 만든다.
- 결과 영역은 화면 아래로 밀리더라도 사용자가 쉽게 확인할 수 있어야 한다.
- 게임형 페이지는 조작 버튼과 게임판이 동시에 보이는지 확인한다.
- 긴 텍스트는 줄바꿈과 overflow 처리를 반드시 확인한다.

## JavaScript 규칙

- 전역 변수는 최소화한다.
- 상태 초기화 함수는 반드시 별도로 둔다.
- 버튼 이벤트는 DOM 로딩 이후 연결한다.
- 같은 버튼에 이벤트가 중복 연결되지 않게 한다.
- `setInterval`, `setTimeout`은 ID를 저장하고 필요 시 해제한다.
- 실행 결과는 화면 표시뿐 아니라 내부 상태까지 확인한다.

## 배포 규칙

- GitHub Pages에서 실제 URL로 접속해 확인한다.
- 브라우저 캐시 때문에 수정이 안 된 것처럼 보일 수 있으므로 강력 새로고침을 한다.
- 모바일 브라우저에서도 직접 접속해 확인한다.
- 한글 파일명 URL은 복사, 공유, 모바일 접속에서 문제가 없는지 확인한다.
- 배포 전후로 파일명이 바뀌면 링크도 함께 확인한다.

## 오늘의 결론

`Youngil_WebApps` 프로젝트의 핵심은 복잡한 프레임워크를 쓰는 것이 아니라, 작은 단일 HTML 웹앱을 안정적으로 반복 제작하는 것이다.  
그러나 단순한 페이지라도 입력 검증, 상태 초기화, 이벤트 중복, 모바일 레이아웃, 타이머 관리, 배포 URL 확인을 놓치면 같은 실수가 반복된다.

앞으로는 새 웹페이지를 만들 때마다 먼저 이 문서를 확인하고, 개발이 끝난 뒤에는 해당 파일명 섹션에 실패와 재발방지를 추가한다.  
이 문서가 누적될수록 `Youngil_WebApps` 프로젝트는 단순한 HTML 모음이 아니라, 반복 가능한 웹앱 제작 하네스로 발전한다.
